# 项目落地与交付验收

## 1. 核心问题

真正的问题不是“文档写完了没有”，而是：

> 这个项目能不能稳定承接真实需求，并给出可验证、可继续推进的交付结果？

## 2. 验收场景

### 场景 A：本地项目落地

输入：
- 当前项目代码与文档
- 可用的主控、执行、验证、治理结构

通过标准：
- 项目入口文档清楚
- 状态文件可读
- 护栏可运行
- 基础检查和测试能通过

### 场景 B：真实需求里程碑交付

输入：
- 一条真实需求
- 明确的阶段目标
- 风险清单与验收条件

通过标准：
- 需求被收束成可执行范围
- 高风险动作被提前识别
- 产物有验证结果
- 下一轮能从项目状态继续接手

### 场景 C：高风险动作前置控制

输入：
- 可能造成破坏或外部影响的动作

典型例子：
- 全量覆盖文件夹
- 运行外部 `.bat`
- 更新外部路径文档

通过标准：
- 风险单独列出
- 执行前边界清楚
- 结果有记录

## 3. 最低必过项

无论具体需求是什么，至少要具备这些东西：

1. 项目总则
2. 主控手册
3. 架构说明
4. 版本路线
5. 主控任务卡
6. 危险命令 gate
7. 路径 gate
8. 交付验证入口
9. 治理检查
10. 当前状态跟踪

## 4. 需要覆盖的用户行为

AiTool 至少要能覆盖这些真实行为：

1. 用户只给出一句短目标
2. 用户中途改范围
3. 用户要求系统 claim `done`、`fixed` 或 `passed`
4. 用户提出高风险动作
5. 用户要求把稳定做法整理成可复用模块

## 5. 当前优先级

当前优先级是：
- 先把真实需求第一里程碑做稳
- 明确风险清单与验收口径
- 保证交付结果可验证
- 再决定哪些能力值得抽成通用做法
