# 桌面便捷工具第一里程碑任务契约

## 1. 任务背景

AiTool 现在以正式项目方式承接第一条真实开发需求。本轮目标不是一次性做完整个商业级桌面工具，而是先把第一里程碑收束清楚，并在真实交付中验证当前 harness 的承接能力。

## 2. 用户正式需求

用户原始需求如下：

1. 文档中转站，双击打开，支持拖动增添入口，拖出复制
2. 文档一键复制，把一个文件夹的内容完全覆盖到另一个文件夹
3. 一键启动导表工具，自定义路径 `F:\F_tunk\工具_指定配置文件导出.bat`
4. 一键更新 svn 路径文档，路径为 `F:\F_tunk\`
5. 根据以上功能模块可以自定义添加相同的模块

## 3. 本轮阶段目标

本轮分为两个阶段：

### 阶段 A：基线收绿

- 确认治理脚本、状态文件、文档与测试重新对齐
- 保留治理层，不通过删治理换绿
- 跑最小检查并回写跟踪状态

### 阶段 B：第一里程碑开发

- 落正式任务契约
- 建立桌面工具基础壳
- 实现文档中转站 MVP
- 为功能 2 / 3 / 4 建立受控入口，不直接放行高风险动作

## 4. 第一里程碑范围

### 本轮必须完成

1. 更新 `.agent/state/active-task.yaml`，把当前任务切到真实需求推进
2. 更新 `.agent/logs/trial-status.md`，持续记录阶段、风险、阻塞和收成
3. 新增本任务契约文档，明确边界、风险与验收
4. 提供可运行的桌面工具骨架
5. 实现文档中转站 MVP：
   - 添加文件或文件夹入口
   - 支持双击打开
   - 支持移除入口
   - 支持把所选入口复制到指定目录
   - 在 Windows 环境下支持拖入增添入口
6. 为高风险功能提供受控入口：
   - 文件夹全量覆盖复制：只做 dry-run
   - 外部 `.bat` 启动：只做路径校验
   - svn 路径文档更新：只做路径校验和规则占位

### 本轮明确不做

1. 文件夹全量覆盖复制的真实执行
2. 外部 `.bat` 的真实启动
3. svn 路径文档的真实更新
4. 文档中转站完整的系统级“拖出复制”
5. 完整插件平台或模块市场
6. exe 打包与安装链路

说明：文档中转站里的“拖出复制”需求，本轮先以 `复制到...` 作为受控替代交互。这样可以先验证真实导出链路，而不会为了系统级拖放细节拖慢整个第一里程碑。

## 5. 高风险动作与控制策略

| 动作 | 风险说明 | 本轮策略 |
| --- | --- | --- |
| 文件夹全量覆盖复制 | 可能覆盖目标目录全部同名内容，破坏面大 | 仅做路径校验、文件统计、重名命中统计和 dry-run 提示 |
| 外部 `.bat` 启动 | 会触发外部副作用，失败路径不透明 | 仅做路径、扩展名、存在性校验 |
| svn 路径文档更新 | 目标文档、写入规则、回滚策略未明确 | 仅做路径可达性校验和规则缺口提示 |

## 6. 第一里程碑验收

本轮通过标准如下：

1. 当前项目基线回绿
2. 中文任务契约落地
3. `trial-status.md` 可持续跟踪
4. 第一里程碑范围明确且未失控扩张
5. 文档中转站 MVP 有真实可运行进展
6. 高风险动作没有被当成普通按钮直接放行

## 7. 验证方式

本轮至少要执行这些验证：

1. `python .\.agent\scripts\check_governance.py --check-registered-paths`
2. `python -m unittest tests.test_phase1_runtime tests.test_governance_contracts tests.test_governance_runtime`
3. 文档中转站与高风险入口相关的最小单元测试
4. 对第一里程碑产物运行一次交付验证脚本并回写状态

## 8. 下一轮准备

如果本轮通过，下一轮优先考虑：

1. 把文件夹全量覆盖复制从 dry-run 推进到受控执行
2. 明确外部 `.bat` 启动的确认流和失败回报
3. 梳理 svn 路径文档更新的输入、输出和回滚规则
4. 评估是否进入 exe 打包链路
