# 版本计划（AiTool）

## 1. 总目标

AiTool 的目标是以正式项目方式承接真实需求，并让 harness 在真实交付中稳定工作。

最终验收不是“还能不能继续讨论部署方式”，而是：

1. 能接住真实需求
2. 能用主控、护栏和验证结构稳定推进
3. 能控制高风险动作
4. 能形成可验证的阶段结果
5. 能把实践中的部署摩擦沉淀成后续可复用的能力

## 2. 当前项目定位

- project owner: `Codex`
- 指定主控线程: `019f3a9b-9dfd-74a0-97eb-820fb8f94ac5`
- AiTool 当前定位：正式项目、harness 实践项目

## 3. 当前主线需求

当前第一条真实需求是：

> 开发一个商业级的桌面便捷工具 exe，界面简约舒适，操作合理，并支持文档中转、文件夹全量覆盖复制、外部导表工具启动、svn 路径文档更新以及后续同类模块扩展。

## 4. 阶段路线

### 阶段 A - 第一层 harness 收口

目标：
- 在 AiTool 中落稳主控、护栏、验证与治理体检

状态：已完成

完成标志：
- Phase 1 harness 已同步
- 治理检查可通过
- 最小测试可通过

### 阶段 B - 真实需求任务契约与第一里程碑

目标：
- 把真实需求翻译成可执行任务契约
- 拆开 `exploratory` 与 `deliverable`
- 收束第一里程碑

关注点：
- 中文任务契约
- 风险清单
- 第一里程碑范围
- 文档中转站 MVP

退出条件：
- 任务契约落地
- 第一里程碑验收清楚
- 高风险动作被显式列出并 gated

状态：当前进行中

### 阶段 C - 基础交互与高风险动作受控化

目标：
- 完成桌面工具基础壳
- 落地文档中转站可用版
- 对高风险动作建立清晰受控入口

重点：
- 文件夹覆盖复制必须有明确风险提示与执行边界
- 外部 `.bat` 启动必须可配置、可校验路径
- svn 路径文档更新必须先定义输入来源与输出规则

### 阶段 D - 打包与模块扩展

目标：
- 形成 exe 打包路径
- 定义同类模块的最小扩展方式

重点：
- 不把“可自定义添加相同模块”过早做成完整插件平台
- 先做可扩，再做重抽象

### 阶段 E - 部署能力沉淀

目标：
- 在真实交付稳定后，总结可复用的部署路径

输出候选：
- 文件包
- skill
- 插件
- 一键部署文档

## 5. 当前重点

当前重点是 **阶段 B**：

1. 用正式项目口径承接真实需求
2. 让中文文档与项目定位一致
3. 收束第一里程碑，不直接冲最终 exe
4. 记录哪些地方仍然阻碍未来的一键部署

## 6. 当前风险

1. 文件夹全量覆盖复制属于高风险动作，不能直接当普通按钮处理
2. 外部 `.bat` 启动依赖路径配置，必须先做路径校验与失败提示
3. svn 路径文档更新目前缺少明确规则，不能直接 claim 可用
4. 一键部署路径仍然需要后续从真实交付中反向沉淀

## 7. 当前不做

当前不做：
- 自动 handoff
- retro 系统
- mode engine
- 多模型 verifier
- 完整 owner/subagent 自动编排
- 第一轮就做完整插件平台
